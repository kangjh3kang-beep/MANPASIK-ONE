# 만파식 개발 철학 (Development Philosophy)

**문서번호**: MPK-PHILOSOPHY-v1.0  
**목적**: 기획·기준 수립과 개발 중 고도화·유기적 대응의 관계를 정의하여, 완벽한 시스템 구축을 추구하는 원칙을 명시한다.

---

## 1. 핵심 원칙: 기준 수립 → 고도화 → 유기적 대응

만파식 프로젝트는 다음 세 단계를 **순차이면서도 반복적으로** 적용한다.

```
[1단계] 기준 수립
  원본 기획안에 완벽 부합 + 추가 제안 반영
  → 세부 기획·계획이 수립되지 않은 모든 항목을 완벽히 수립
  → "이 기준부터 개발한다"는 명확한 베이스라인 확정

[2단계] 개발 중 수정·보완·고도화
  구현 과정에서 기준을 수정·보완하며 시스템을 개선
  → 품질 게이트(Level 1/2/3), 코드 리뷰, 테스트로 완결성 검증
  → 발견된 개선점은 문서·코드에 반영하고 다음 단계에 전달

[3단계] 예상치 못한 사항에 유기적 대응
  개발 과정에서 생각지 못한 이슈·요구·제약이 발생하면 유기적으로 대응
  → KNOWN_ISSUES, CHANGELOG, 보완사항 목록에 기록
  → 기획·설계 문서를 보완하여 "새 기준"으로 정리
  → 원본 의도(홍익인간, 차동측정, 오프라인 100%, 규제 준수)는 유지
```

**목표**: 기준이 있는 상태에서 개발을 시작하고, 개발을 통해 기준을 계속 다듬어 **완벽한 시스템 구축 및 완성**을 추구한다.

---

## 2. 기준 수립(1단계)에서 할 일

- **원본 기획안(v1.0)** 과 **발전 수립안(plan-original-vs-current-and-development-proposal.md)** 에 따라, 아직 세부 수립이 안 된 항목을 **완벽히** 정리한다.
- 다음을 포함한다(예시).
  - 데이터 패킷 표준(패밀리C), 사이트맵·스토리보드 공식 문서
  - MSA 확장 로드맵, AI 에이전트–Phase 매핑
  - 기획–구현 추적성 매트릭스, tenant_id·audit_logs 등 스키마·정책
- "이 문서/이 표가 현재 기준이다"라고 참조할 수 있도록 **단일 문서 또는 경로**를 두고, `.cursor/rules`·CONTEXT·AGENTS에서 이 기준을 참조한다.

---

## 3. 개발 중 고도화(2단계)에서 할 일

- **Quality Gate(QUALITY_GATES.md)** 에 따라 매 작업(Level 1), Stage 완료(Level 2), Phase 완료(Level 3) 시 검증한다.
- 구현·리뷰·테스트 과정에서 나온 **개선안·보완안**은:
  - 코드·설계에 반영하고,
  - 해당하는 기획/설계 문서를 **수정·보완**하여 "새 기준"으로 갱신한다.
- CHANGELOG·CONTEXT·KNOWN_ISSUES에 **무엇을 바꿨는지, 왜 바꿨는지**를 기록하여, 다음 담당자(AI·인간)가 이어받을 수 있게 한다.

### 3.1 우회·미루기 금지 원칙 (No Deferral/Workaround by Default)

- **원칙**: 구현 항목을 "나중에 하자", "우회해서 가자"로 **미루거나 우회하는 것을 기본 옵션으로 두지 않는다.** 막히면 해결점을 찾아 **정상적으로 구현·구축**한다.
- **이유**: 미루기·우회가 쌓이면 기술 부채와 불완전한 동작이 누적되고, 의료기기·엔터프라이즈 품질과 규제 추적성에 악영향을 준다.
- **실무**:
  - 구현 중 장애(빌드 실패, 의존성 이슈, 스펙 불명확 등)가 나오면 **원인 파악 → 대안 탐색 → 해결 후 정상 구현**을 우선한다.
  - 부득이하게 **일시적 우회 또는 연기**가 필요한 경우에만 예외를 둔다. 이때는 반드시 **KNOWN_ISSUES에 등록**하고, **해결 조건·목표 시한**을 명시하며, 주기적으로 해결 시도를 한다. "TODO/나중에"만 남기고 이슈로 남기지 않는 것은 금지에 가깝게 피한다.

---

## 4. 예상치 못한 사항에 대한 유기적 대응(3단계)

- **생각지 못한 이슈**(버그, 성능, 규제 해석, 보안, UX, 하드웨어 제약 등)가 발생하면:
  1. **즉시 기록**: KNOWN_ISSUES, CHANGELOG (발생 이슈·해결·우회).
  2. **원본 의도 유지**: 홍익인간, 차동측정 공식, 오프라인 100%, 5개국 규제 준수, 의료기기 품질 등 **핵심 가치와 충돌하지 않는 범위**에서 대응한다.
  3. **기준 갱신**: 대응 결과가 설계·동작·정책에 영향을 주면, 해당 기획·설계 문서를 **보완**하고, 필요 시 기획안 v1.1(또는 부록)에 "변경 이력·예외·판단 근거"를 남긴다.
- **일회성 우회**가 아닌 **구조적 변경**은, 가능하면 기획–구현 추적성 매트릭스와 연결하여 "어떤 요구/이슈 때문에 이렇게 바꿨는지" 추적 가능하게 한다.

---

## 5. 요약

| 단계 | 내용 |
|------|------|
| **기준 수립** | 원본 + 제안 반영 → 미수립 세부 기획·계획 완전 수립 → 개발 베이스라인 확정 |
| **고도화** | 개발 과정에서 수정·보완·개선 → Quality Gate로 검증 → 문서·코드 갱신 |
| **우회·미루기 금지** | 구현은 정상 구현 우선; 우회/연기는 예외이며 KNOWN_ISSUES + 해결 조건·시한 필수 |
| **유기적 대응** | 예상치 못한 이슈 → 기록·대응·기준 갱신 → 핵심 가치 유지 |

**결론**: "기준을 먼저 완벽히 세우고, 개발하면서 수정·보완·고도화하며, 생각지 못한 일에는 유기적으로 대응해 완벽한 시스템 구축 및 완성을 추구한다"는 접근이 맞으며, 본 문서는 이를 프로젝트의 공식 개발 철학으로 정의한다.
